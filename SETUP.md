## 1. SQL Server 19 (SSMS) — Connection Details

| Setting              | Value                                     |
|----------------------|-------------------------------------------|
| Server Name          | `localhost` or `.\SQLEXPRESS`              |
| Authentication       | **SQL Server Authentication (Recommended)** OR **Windows Authentication** |
| Port                 | `1433` (default TCP)                       |
| Database Name        | `JRMSUKatipunanCampusLibrary`             |
| User / Password      | `JRMSUKatipunanCampusLibrary` / `JRMSUKatipunanCampusLibrary` (for SQL Auth) OR N/A (for Windows Auth) |
| ODBC Driver          | `ODBC Driver 17 for SQL Server`            |

### How Django connects

Django reads from a `.env` file generated by `setup_db.py`. If no `.env` file exists, it defaults to MSSQL with Windows Authentication. The supported engines are:

| Engine | `.env` value | Driver to install |
|--------|-------------|-------------------|
| **SQL Server (SSMS 19) (RECOMMENDED)** | `mssql` | `pip install mssql-django pyodbc` |
| MySQL / MariaDB (XAMPP) (OPTIONAL) | `mysql` | `pip install mysqlclient` (or `pymysql`) |
| PostgreSQL (OPTIONAL) | `postgresql` | `pip install psycopg2-binary` |

> **Note:** If your SQL Server instance name is different (e.g., `.\SQLEXPRESS`), run `setup_db.py` and enter the correct host.
> **For SQL Authentication:** Open your `backend/.env` file and set `DB_WINDOWS_AUTH=False`, `DB_USER=JRMSUKatipunanCampusLibrary`, and `DB_PASSWORD=JRMSUKatipunanCampusLibrary`.

## 2. Database Setup (Interactive)

Run the interactive database setup script **once** before starting Django:

```
cd backend
python setup_db.py
```

This script will:
1. Let you choose your database engine (**SSMS 19 is recommended**; MySQL/PostgreSQL are optional).
2. Prompt you for connection details (host, port, credentials).
3. Generate a `.env` file that `settings.py` reads automatically.
4. Auto-create the database if the driver is installed.

> **Where is the `.env` file?**
> The `setup_db.py` script will create a file named `.env` located inside the `backend/` directory (`backend/.env`). You can open this file in your code editor anytime to manually update your database connection credentials.

### Alternative: Manual Database Creation

#### Option A: Via SSMS GUI

1. Open **SQL Server Management Studio 19**.
2. Connect to your server instance (`localhost` or `.\SQLEXPRESS`).
3. Right-click **Databases** in Object Explorer.
4. Click **New Database...**
5. Database name: `JRMSUKatipunanCampusLibrary`
6. Click **OK**.

#### Option B: Via the legacy helper script

```
cd backend
python create_db.py
```

This runs a `pyodbc` script that creates the database if it doesn't already exist.

## 3. Backend — Django Setup

Open a terminal at the **project root**:

```
cd backend
```

### 3a. Create and activate virtual environment (first time only)

```
python -m venv venv
venv\Scripts\activate
```

You should see `(venv)` in the prompt.

### 3b. Install Python dependencies

```
pip install django djangorestframework django-cors-headers mssql-django django-filter pyodbc python-dotenv
```

### 3c. Run database migrations

Django uses a migration system to create and update database tables. You must run migrations whenever models change.

**Generate migration files** (detects model changes and creates migration scripts):
```
python manage.py makemigrations Api
```

**Apply migrations to the database** (creates actual tables in SQL Server):
```
python manage.py migrate Api
python manage.py migrate
```

Or use the architecture wrapper commands:
```
python manage.py add_migration
python manage.py update_database
```

### 3d. Seed CMS data from asset files

```
python manage.py seed_assets
```

### 3e. Create an admin superuser (optional)

```
python manage.py createsuperuser
```

### 3f. Run the backend server (port 8000)

```
python manage.py runserver 8000
```

Backend loads at: **http://localhost:8000**

## 6. Migration Cheat Sheet

| Task                         | Command                                   |
|------------------------------|-------------------------------------------|
| Detect model changes         | `python manage.py makemigrations`         |
| Apply migrations to DB       | `python manage.py migrate`                |
| Show migration status        | `python manage.py showmigrations`         |
| Rollback a migration         | `python manage.py migrate Features 0001`  |
| Create empty migration       | `python manage.py makemigrations --empty Features` |
| Architecture wrapper (make)  | `python manage.py add_migration`          |
| Architecture wrapper (apply) | `python manage.py update_database`        |

> **Important:** Always run `makemigrations` first to generate the migration files, then `migrate` to apply them. Never skip `makemigrations`.

## 8. Troubleshooting

| Symptom | Fix |
|---------|-----|
| `'python' is not recognized` | Install Python 3.14+ and add to PATH |
| `'npm' is not recognized` | Install Node.js 20+ and add to PATH |
| `'venv' is not recognized` | Run `python -m venv venv` inside `backend/` |
| `ModuleNotFoundError` | `pip install -r requirements.txt` or `pip install <module>` |
| `CORS error in browser` | Ensure Django runs on port 8000 |
| Images not loading | Django server must be running |
| `pyodbc.InterfaceError: ODBC Driver not found` | Install ODBC Driver 17 for SQL Server |
| `django.db.utils.OperationalError` | Ensure SQL Server service is running in Windows Services |
| Migration issues | Delete old `db.sqlite3` if it exists, re-run `makemigrations` then `migrate` |
| `Login failed for user` | Ensure SSMS uses Windows Authentication and `Trusted_Connection=yes` is set |

## 12. Flow Chains — Core Principles

To ensure architecture consistency, these chains must **always** be followed.

### 12A. Frontend Flow Chain

```
Pages
  ↓  (route-level composition only)
Features
  ↓  (business logic and workflows)
Hooks / State / API (Endpoints)
  ↓  (reusable logic, shared state, backend calls)
Shared Components
  ↓  (presentational primitives only)
Libs / Utilities
  ↓  (infrastructure: auth, clients, helpers)
Assets
```

### 12B. Backend Flow Chain

```
Models/New Models Added or created
  ↓
Enums
  ↓
Django ORM
  ↓
Repository Implementation
  ↓
Repository Interface
  ↓
Service Implementation
  ↓
Service Interface
  ↓
Helpers
  ↓
API Controllers
  ↓
Middleware
  ↓
manage.py → settings.py → Custom Management Commands
```

### 12C. HTTP Request Flow Chain

```
Incoming Request
  ↓
Django Middleware  (rate limit, CSRF, auth guards, idempotency)
  ↓
API Controller    (parse request, call service, return response)
  ↓
Service Layer     (validate, enforce rules, orchestrate)
  ↓
Repository Layer  (query, persist, filter)
  ↓
Database          (final state)
```

### 12D. Cache Read Flow

```
Request → Middleware → Controller → Service → Cache Service
  Cache HIT?  → Return Data
  Cache MISS? → Repository → Database → Store in Cache → Return Data
```

### 12E. Cache Write / Invalidation Flow

```
Request → Controller → Service → Repository → Database
  Success? → Invalidate: book:{id}, book:list, dashboard:stats → Return Result
```

### 12F. Contact Form Flow (Domain-Specific)

```
Visitor submits Contact Form
  ↓
Django Middleware (rate_limit_middleware, CSRF)
  ↓
ContactController  POST /api/contact
  ↓
ContactService  (validate, sanitize via input_sanitizer, send email notification)
  ↓
ContactRepository  (persist inquiry)
  ↓
Database  ContactMessages table
```
